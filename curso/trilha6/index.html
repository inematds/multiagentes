🧭 Fundamentos visuais

A topologia básica de uma equipe, a diferença em relação a subagentes e a árvore de decisão de quando aplicar.

O que mostra:

Diagrama "Você → Orquestrador → Agente A/B/C/D" com setas de mailbox bidirecional entre os teammates. Os agentes não conversam direto com você; conversam pelo orquestrador e entre si por mensagens.

Como aplicar:

Quando alguém perguntar "como funciona Agent Teams?", esta é a primeira imagem que você desenha. Fixa a hierarquia e o canal de comunicação.

O que mostra:

Lado esquerdo: agente principal cria N subagentes paralelos, cada um devolve um resultado isolado. Lado direito: líder distribui tarefas para integrantes que se comunicam entre si via lista compartilhada.

Como aplicar:

Pergunta-chave: "preciso só de respostas paralelas ou de coordenação?" Se for paralelo puro, subagent. Se precisa de mailbox e contexto compartilhado, equipe.

O que mostra:

Tabela em 3 colunas: Use Teams (3+ áreas, pesquisa paralela, colaboração com React, quality gates). Evite Teams (sequencial 1→2→3, contexto único, mesmos arquivos, tarefa simples). Subagentes (resultado focado, sem cross-talk, preservar tokens).

Como aplicar:

É a árvore de decisão antes de qualquer spawn. Se sua tarefa cair na coluna do meio, você está prestes a queimar tokens à toa.

⚙️ Configuração & prompting

Como ligar Equipes no settings.json, o template de prompt que sempre funciona e a tabela FAÇA / NÃO FAÇA.

O que mostra:

Bloco JSON em settings.json com "env": { "AGENT_TEAMS": 1 } à esquerda; à direita, status "Equipes Ativas ✓" com 3 agentinhos ligados.

Como aplicar:

Sem essa flag, qualquer prompt sobre "criar equipe" cai em fluxo de subagent comum. Confira no início de toda sessão nova.

O que mostra:

Esqueleto: "Crie uma equipe de [N] usando [modelo]…" seguido de 1–N linhas "[Papel] → possui + produz". Três exemplos derivados (Designer de API → /api → rotas + spec; Engenheiro de BD → migrações + esquema; Redator de Testes → testes + suíte).

Como aplicar:

Sempre que for spawnar uma equipe, escreva o prompt nesse formato literal. "Possui" define território (arquivos), "produz" define entregável (mailbox).

O que mostra:

Tabela espelhada. FAÇA: defina responsáveis por arquivo, defina entregáveis, nomeie destinatários, 3–5 integrantes, dê contexto completo. NÃO FAÇA: compartilhar mesmo arquivo, entregáveis vagos, pressupor o plano, 10+ integrantes, sem contexto prévio.

Como aplicar:

Use como checklist antes de enviar o prompt. Cada linha do "NÃO FAÇA" é um bug que aparece nas Trilhas 4 (diagnóstico) — esse card os previne de origem.

🎮 Regras, contexto & plano

As 3 regras de ouro, o que cada teammate efetivamente "sabe" e o ciclo de aprovação de plano.

O que mostra:

Três blocos numerados: (1) Tenha seu território — um arquivo, um dono. (2) Mensagens diretas — formato API, evite líder central. (3) Comece em paralelo — sequencial é antipattern, todos começam agora.

Como aplicar:

Antes de aceitar qualquer plano sugerido pelo orquestrador, faça o pente-fino com essas 3 regras. Quebrar uma delas é causa raiz de 80% das falhas.

O que mostra:

Esquerda (vermelho): "Sem contexto" — histórico de chat bloqueado, sem prompt. Direita (verde): "Permissões OK" — mesmas permissões (Bash, Editar, Escrever) e auto-load de CLAUDE.md, servidores MCP e Skills.

Como aplicar:

Cada teammate começa cego sobre a conversa anterior. Tudo que ele precisa saber tem que vir no prompt de spawn ou no CLAUDE.md auto-carregado.

O que mostra:

4 estágios numerados: (1) Pesquisa — somente leitura. (2) Enviar plano. (3) Líder revisa — pode pedir nova revisão. (4) Executar — só aqui o teammate ganha acesso total de escrita.

Como aplicar:

É o gate que separa "agente que toca os arquivos sem pensar" de "agente revisado". Para qualquer mudança não-trivial, force plan mode antes do execute.

🛟 Operação saudável

Os erros mais comuns e suas correções, consciência de custos e o caminho de encerramento limpo.

O que mostra:

Tabela "Erro comum → Correção": pedir permissões → pré-aprovar; conflitos de arquivo → atribuir donos; agente ocioso → trabalho independente; muitos tokens → menos agentes; trabalho perdido → arquivos temporários; aprovação incorreta → revisão humana.

Como aplicar:

Quando algo der errado em produção, esta é a tabela de triagem. Identifique o sintoma da esquerda e aplique a correção da direita.

O que mostra:

Topo: 1 agente sozinho = custo 1x; 3 agentes ≈ 3x; 5 agentes ≈ 5x. Embaixo, 4 dicas: máx. 3 por vez; mantenha em paralelo; use subagentes (resultado-only); desligue cedo.

Como aplicar:

Custo de equipe é multiplicativo, não aditivo na percepção. Antes de spawnar 5, pergunte: "isso vale 5x o custo do solo? a wall-clock economizada compensa?"

O que mostra:

Esquerda (verde): caminho limpo — colegas salvam, equipe de limpeza limpa ~/.claude/team/, fechar sessão. Direita (vermelho): forçar — arquivos órfãos, configs órfãs, estado desconhecido. Rodapé: "Salve primeiro, limpe tudo, depois feche".

Como aplicar:

Encerrar uma equipe é tão importante quanto criá-la. Sempre passe pelos 3 passos antes de fechar a sessão — Ctrl+C cego deixa rastros que quebram a próxima execução.
